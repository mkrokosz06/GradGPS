The fixes are sequenced by dependency. Most of the High/Critical findings
collapse into ONE auth rebuild, and a few things are done first because they are
cheap and unblock the rest.


--------------------------------------------------------------------------------
THE ONE DECISION TO MAKE FIRST
--------------------------------------------------------------------------------
Everything hinges on what "real authentication" looks like.

The backend .env already stubs a GOOGLE_CLIENT_ID, so the intended path was
almost certainly Google Sign-In (PSU issues Google Workspace accounts to every
student — a natural fit). This plan assumes that.

  RECOMMENDATION: Google OIDC.

Alternative: email + password — means you own password hashing, reset flows, and
email verification (more surface, more liability). If you choose this instead,
only Phase 1 changes; the rest of the plan is unaffected.

--------------------------------------------------------------------------------
PHASE 1 — Real authentication (THE CORE)
Fixes Criticals #1, #2 and Highs #7, #9, plus #13
--------------------------------------------------------------------------------
The big one. Everything else is downstream of "identity is a spoofable header."

BACKEND:
  [ ] 1. Verify tokens in deps.py:
         Rewrite get_user_id into a dependency that reads
         "Authorization: Bearer <token>", verifies the Google ID token
         (signature, aud = client ID, exp), and returns the `sub` claim as the
         canonical user_id. Never trust a client-supplied identity again.

  [ ] 2. Decouple user_id from email — users.py:30-37:
         Stop deriving IDs from email (kills enumeration/squatting, finding #12).
         user_id becomes the verified Google `sub`. POST /users/create goes away
         or becomes an authenticated "upsert my profile from my token" call.

  [ ] 3. Sanitize the S3 key — transcript.py:192:
         Even with a verified sub, validate against ^[A-Za-z0-9_-]+$ before using
         it in the object key (finding #9).

  [ ] 4. Gate /admin/* — admin.py:
         Add an auth dependency plus an admin allowlist (env var of admin subs).
         Currently wide-open to anonymous callers — second priority only to the
         header fix.

MOBILE:
  [ ] 5. Wire Google Sign-In:
         Add expo-auth-session / Google provider; signIn() in AuthContext.tsx
         exchanges for an ID token instead of taking a raw uid.

  [ ] 6. Store the token in SecureStore — AuthContext.tsx:40:
         Move the token to expo-secure-store (Keychain/Keystore), not
         AsyncStorage (finding #13). Name/email can stay in AsyncStorage.

  [ ] 7. Send Authorization: Bearer:
         Update services/api.ts and every *Service.ts to send the token header
         instead of x-user-id.

PHASE 2 — Upload hardening (Highs #4, #5)
--------------------------------------------------------------------------------
transcript.py:141-149:

  [ ] 1. Reject before reading: check Content-Length and enforce a max
         (e.g. 5 MB) -> 413.

  [ ] 2. Validate file.filename is not None, then verify the %PDF- magic bytes
         on the read content — not just the .endswith(".pdf") string.

  [ ] 3. Cap parsed pages in transcript_parser.py (e.g. pdf.pages[:50]) and wrap
         parsing in a timeout to defang PDF bombs.


--------------------------------------------------------------------------------
PHASE 3 — DoS / rate limiting (High #6)
--------------------------------------------------------------------------------
  [ ] 1. Add slowapi (or equivalent) with per-user/per-IP limits, tightest on
         /audit, /timeline, /transcript/upload, and the scan-heavy /courses/*.

  [ ] 2. courses.py:39-54: the full-table-scan-on-cache-miss + outbound
         RMP/bulletin calls are the worst DoS lever — add a timeout budget on the
         outbound scrapes and consider persisting the course cache so distinct-
         code loops can't repeatedly force scans.

--------------------------------------------------------------------------------
SUGGESTED ORDER & WHY
--------------------------------------------------------------------------------
Phase 0 first (fast, removes free wins + CVE deps), then Phase 1 as the main
event — it's the load-bearing fix and touches both backend and mobile, so it's
the longest. Phases 2-4 are self-contained and can follow in any order, though
Phase 2 is a good next step since it's small and closes an unauthenticated DoS.

Nothing here changes local dev workflow until Phase 1's mobile changes land.
Keep a dev bypass (env-gated) so you can still run against local without Google
Sign-In.

Rough effort:
  Phase 0    ~30 min
  Phase 1    the bulk (a full session: backend + mobile + testing Google flow)
  Phases 2-4 ~1-2 hours each
